# AGENTS.md — guide for AI assistants working on this repo

## What this project is

A Safari Web Extension (wrapped in macOS and iOS/iPadOS app targets) that redirects DuckDuckGo searches to the user's self-hosted SearXNG instance — the same approach used by apps like Braver Search, since Safari does not allow custom search engines. The user sets DuckDuckGo as Safari's search engine; a content script intercepts the search before the page loads and redirects to `<instance>/search?q=<query>`.

## Layout & source of truth

```
extension/                                   ← SOURCE OF TRUTH for the web extension
  manifest.json                                MV3, content script on duckduckgo.com
  redirect.js                                  the actual redirect logic (document_start)
  popup.html / popup.js                        settings UI (instance URL + enable toggle,
                                               stored in browser.storage.local)
  images/                                      icons (generated programmatically)
SearXNG for Safari/                          ← Xcode project (generated by Apple's
                                               safari-web-extension-converter; Swift code
                                               is converter boilerplate, rarely touched)
  Shared (Extension)/Resources/              ← BUNDLED COPY of extension/ — never edit
  SearXNG for Safari.xcodeproj
sync-extension.sh                            ← purges Syncthing sync-conflicts then
                                               copies extension/ → Resources/ (rsync --delete)
```

**Critical rule:** edit the extension only in `extension/`, then run `./sync-extension.sh`. Editing `Shared (Extension)/Resources/` directly will be silently overwritten by the next sync.

## Build & install (macOS)

```sh
./sync-extension.sh
cd "SearXNG for Safari"
xcodebuild -scheme "SearXNG for Safari (macOS)" -configuration Release \
  -derivedDataPath build \
  DEVELOPMENT_TEAM=AWFP66P6U2 CODE_SIGN_STYLE=Automatic \
  MACOSX_DEPLOYMENT_TARGET=12.0 build
rm -rf "/Applications/SearXNG for Safari.app"
ditto "build/Build/Products/Release/SearXNG for Safari.app" "/Applications/SearXNG for Safari.app"
```

- Signing team `AWFP66P6U2` is the user's Apple Development certificate ("Michele Belleri"). Properly signed builds work in Safari without the *Develop → Allow Unsigned Extensions* toggle; ad-hoc (`CODE_SIGN_IDENTITY="-"`) builds do not persist across Safari restarts.
- `MACOSX_DEPLOYMENT_TARGET=12.0` overrides the project's stale `10.14` setting (Xcode 27 refuses targets below 12.0). Keep it on the command line — don't bake it into `project.pbxproj` (the converter-file rule below still applies).
- `rm -rf` of the old `/Applications/SearXNG for Safari.app` is mandatory: `ditto` does not delete destination files missing from the source, so any orphans from the previous install (Syncthing junk, Swift dylibs from a different deployment target) will survive and break the new bundle's code-signing seal. See gotcha #3.
- iOS builds use the `SearXNG for Safari (iOS)` scheme and must be deployed to a device from Xcode.

## Signing prerequisites (run once per machine)

`xcodebuild` will fail with `No "Mac Development" signing certificate ... with a private key was found` if any of these slip:

1. **`xcode-select` points at full Xcode**, not Command Line Tools:
   ```sh
   xcode-select -p   # expect /Applications/Xcode*.app/Contents/Developer
   sudo xcode-select -s /Applications/Xcode.app/Contents/Developer
   ```
2. **License accepted** after every Xcode upgrade: `sudo xcodebuild -license accept`.
3. **Code-signing identity present**: `security find-identity -v -p codesigning` must list `Apple Development: Michele Belleri (J4WXNEXWW5)` as **valid**.
   - `0 valid identities found` → either no cert or broken trust chain.
   - `1 identities found` but with `CSSMERR_TP_NOT_TRUSTED` → cert is there but the **Apple WWDR intermediate** is missing/outdated. Install WWDR G3 alongside whatever else is in the keychain:
     ```sh
     curl -fsSL https://www.apple.com/certificateauthority/AppleWWDRCAG3.cer -o /tmp/WWDRG3.cer
     security import /tmp/WWDRG3.cer -k ~/Library/Keychains/login.keychain-db
     ```
   - Cert truly missing → Xcode → Settings → Accounts → Manage Certificates → **+** → Apple Development.
4. **Xcode Accounts may label `AWFP66P6U2` as "Personal Team"** even when it is the paid program. Don't conclude the program lapsed from the UI label alone — check the cert's `OU=` (e.g. `openssl x509 -text` on the cert in the keychain).

## Known gotchas

1. **Bundle ID prefix validation**: the app target must be `it.micky.searxng-safari` and the extension `it.micky.searxng-safari.Extension`. If they don't share the prefix, the build fails at `ValidateEmbeddedBinary`. (The converter once generated mismatched IDs — check `PRODUCT_BUNDLE_IDENTIFIER` in `project.pbxproj` if this error appears.)
2. **Do not re-run `safari-web-extension-converter`** unless absolutely necessary: it regenerates the whole `SearXNG for Safari/` folder, losing any manual project fixes.
3. **Syncthing `*.sync-conflict-*` files silently break code signing.** This directory is synced by Syncthing, which produces `.syncthing.*` and `*.sync-conflict-*` files at any time. They are gitignored, but if any survive into the built `.app`/`.appex` they invalidate the seal and Safari **silently** refuses to load the extension — no error in Safari Settings → Extensions, the entry just never appears. Diagnostic:
   ```sh
   spctl -a -vv "/Applications/SearXNG for Safari.app"
   # → "a sealed resource is missing or invalid"   ← smoking gun
   codesign --verify --deep --strict --verbose=2 "/Applications/SearXNG for Safari.app"
   # → "file added: ...sync-conflict..."           ← same root cause
   ```
   `sync-extension.sh` now purges them from the whole tree before each sync. If you see the `spctl` error on an already-installed bundle, `rm -rf` it and reinstall — purging in place is not enough because the *signature* still embeds the bad seal.
4. **Extension settings persist** across rebuilds (stored in `browser.storage.local`, keyed by the extension bundle ID) — as long as the bundle ID doesn't change.
5. After reinstalling the app, Safari may require re-enabling the extension in Settings → Extensions, and the toolbar icon may need to be added via *View → Customize Toolbar*.
6. **`spctl rejected` ≠ broken bundle.** For Apple Development-signed (non-notarized) builds, `spctl -a` always reports `rejected`. That alone is fine — Safari doesn't use Gatekeeper to gate web extensions. The authoritative check is `codesign --verify --deep --strict`. Only worry if it reports a real failure ("sealed resource is missing or invalid", etc.).

## Conventions

- Code, UI strings, commit messages, README: **English**. Conversation with the user: **Italian**.
- Keep the extension minimal — no analytics, no external requests, no frameworks. Plain JS/HTML/CSS only.
- License: MIT. Repo: https://github.com/TinyWorkshopDesign/searxng-for-safari (GitHub account `TinyWorkshopDesign`).
- `build/` directories and `.claude/` are gitignored; don't commit build products.

## Testing a change

1. `./sync-extension.sh`, rebuild, `rm -rf` old install, `ditto` new bundle.
2. Verify signature integrity before launching: `codesign --verify --deep --strict --verbose=2 "/Applications/SearXNG for Safari.app"` must say `valid on disk` and `satisfies its Designated Requirement`. If not, see gotcha #3.
3. In Safari (with DuckDuckGo as default search engine and the extension enabled + URL configured), search from the address bar and confirm you land on the SearXNG instance with the query intact.
4. Test the popup: URL validation (rejects non-http(s)), Save feedback, Enabled toggle actually stops the redirect.
